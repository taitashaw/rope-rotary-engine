# rope-rotary-engine

**Open-source hardware Rotary Position Embedding (RoPE) unit, in production-grade SystemVerilog.**

RoPE is the positional encoding every frontier open-weight model uses — Llama 3/4, DeepSeek V3, Qwen 2/3, Mistral, Gemma. In software it's one `torch.einsum`. In hardware, nobody had open-sourced a reference RTL for it. This is that RTL.

| Metric | Value |
|---|---|
| Correctness | **16 / 16 directed tests, 0 bit-errors** against a Python golden model |
| Accuracy vs float64 | `max_abs_err = 5.58e-05` (< 2 Q1.15 LSBs) |
| Target clock | 400 MHz on Zynq UltraScale+ (xczu7ev-2) — see [note on timing](#note-on-timing) |
| Throughput (design target) | 128 bits / cycle = **51.2 Gbps** per engine |
| Datapath | Q1.15 signed, saturating |
| Pipeline latency | 5 cycles end-to-end |
| Resource footprint (design target) | ~16 DSP48E2 + ~114 BRAM36 + ~1.2k LUT / ~1.5k FF |
| Config | `HEAD_DIM=128`, `MAX_SEQ_LEN=2048`, `PAIRS_PER_CYCLE=4` (parameterized) |
| License | Apache-2.0 |

The engine is interface-compatible with the AXI4-Stream convention used by [flashattn-softmax-engine](https://github.com/taitashaw/flashattn-softmax-engine) and [kvcache-compress-engine](https://github.com/taitashaw/kvcache-compress-engine). Chain the three and you have the full open-source hot path for transformer inference.

## Quick start

```bash
# 1. Clone
git clone https://github.com/taitashaw/rope-rotary-engine.git
cd rope-rotary-engine

# 2. Install simulator (one time)
sudo apt-get install -y iverilog gtkwave python3-numpy

# 3. Run the self-checking testbench
make sim
# => RESULT: PASS (16/16 tests, 0 bit-errors)

# 4. Open the waveform
make waves
```

That's it. `make sim` regenerates test vectors from the Python golden, builds the RTL, runs the testbench, and checks every output beat bit-for-bit against the golden.

## What it does

Given a query/key vector `x` of dimension `HEAD_DIM` at sequence position `m`, it computes the RoPE rotation:

```
    pair i = (x[2i], x[2i+1]),   i ∈ [0, HEAD_DIM/2)
    x'[2i]   = x[2i] * cos(m·θ_i) − x[2i+1] * sin(m·θ_i)
    x'[2i+1] = x[2i] * sin(m·θ_i) + x[2i+1] * cos(m·θ_i)
```

…with `θ_i = base^(−2i/HEAD_DIM)`, `base = 10000` (Llama-family default; configurable via LUT regeneration for long-context models).

One beat of the streaming interface carries `PAIRS_PER_CYCLE = 4` pairs. A full `HEAD_DIM = 128` token therefore streams in 16 beats.

## Simulation

The testbench drives 16 golden-generated vectors (including edge cases: all-max-positive, all-max-negative, all-zero, alternating) at four positions spanning the LUT range and checks every output beat bit-for-bit.

```bash
make sim
# -------------------------------------------------------
# rope_engine_tb: HEAD_DIM=128 PAIRS/CYC=4 TESTS=16
# -------------------------------------------------------
# TOKENS_COLLECTED = 16 / 16
# ERRORS           = 0
# RESULT: PASS (16/16 tests, 0 bit-errors)
# -------------------------------------------------------
```

A VCD is written to `tb/vectors/rope_engine_tb.vcd`. Open with GTKWave:

```bash
make waves
```

Suggested signals to watch first: `dut.s_valid`, `dut.s_position`, `dut.eff_position`, `dut.cos_arr[0]`, `dut.sin_arr[0]`, `dut.m_valid`, `dut.m_data`.


### Note on timing

The 400 MHz number in the summary table is a **design target** based on DSP48E2 speed-grade and pipeline depth — not a post-route measurement. Actual WNS depends on your device, speed grade, floorplan, and BRAM placement. Run `make synth` on your hardware and read `build/rope_engine_timing.rpt` for the ground truth.

## Block design + bitstream

For a bootable end-to-end demo on a ZCU104/ZCU106, the included block design wires a Zynq PS + AXI-DMA + SmartConnect + `rope_engine`:

```bash
make bd
# build/rope_soc/rope_soc.runs/impl_1/rope_soc_wrapper.bit
```

Topology (see [docs/architecture.md](docs/architecture.md) for the full Mermaid diagram):

```
   [ PS DDR ] ⇄ [ S_AXI_HP0 ] ⇄ [ AXI-DMA MM2S/S2MM ] ⇄ [ rope_engine ]
        ^
        │ control
   [ PS M_AXI_HPM0 ] → [ SmartConnect ] → [ AXI-DMA AXI-Lite ]
```

PS software (not in this repo) allocates an input buffer in DDR, programs MM2S with the token address + TUSER=position, kicks off the DMA, and the S2MM picks up the rotated tokens back into DDR.

## Design choices worth the read

- **Interleaved pairs, not rotated halves.** Matches the original RoFormer paper and the Llama-family reference implementation. Models that use the rotated-halves convention need a byte-shuffle in front of the engine — see `docs/roadmap.md`.
- **Q1.15 over BF16.** Every open-weight model publishes activations that fit comfortably in ±1 after RMSNorm. Q1.15 gives deterministic bit-exact behavior and maps directly to DSP48E2 primitives. BF16 is on the roadmap for models with outlier-heavy activations.
- **Banked LUTs, not CORDIC.** At `MAX_SEQ_LEN = 2048` the LUT fits in 114 BRAM36 on xczu7ev with no penalty to timing. CORDIC saves BRAM but costs 10+ LUTs per rotation and adds iteration latency — a worthwhile tradeoff on BRAM-constrained devices, listed on the roadmap.
- **Banked, not broadcast.** One LUT instance per PE, storing only the pair indices that PE services. Cuts BRAM 4× vs a naive shared full-depth LUT and keeps the placer happy.
